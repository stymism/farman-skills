# handoff

今の会話を圧縮して、別のセッション(別のチャット)が読めばすぐ続きを始められる「引き継ぎ文書」にするスキル。

## なぜ使うのか

長い会話は文脈が積み上がる。話題が別の領域に移ったり(例: 健康の話 → もし収入が変わったらの家計試算 → 資産運用の話)、いったん区切って後で続きをやりたくなったりする。そのとき、会話をまるごと持ち運ぶのは重いし、恒久的に残したい事実は自分のメモやノートに書けばよいが、細かい経緯や一時的な検討内容まで残すとノイズになる。

引き継ぎ文書は「この会話の続きをするためだけに必要な情報」を一時ファイルに切り出す仕組み。恒久メモを汚さずに、別セッションへ作業を渡すために使う。

## このスキルがやること(全体像)

1. どこに引き継ぐか(別の領域・別のチャット)と、次のセッションで何をするかを把握する
2. 今の会話を、続きを始めるのに必要な情報だけに圧縮して引き継ぎ文書にまとめる
3. Markdown ファイルとして保存する
4. 保存したファイルパスと、引き継ぎ先での始め方を伝える

## 手順

### 1. 引き継ぎ先と目的を確認する

- 発言から、どこに/何のために引き継ぐかが明確なら確認不要でそのまま進める。
- 不明確な場合のみ、引き継ぎ先と「次のセッションで何にフォーカスするか」を1回だけ尋ねる。
- 起動フレーズに続けて目的が渡されていたら(例: 「家計の試算に引き継ぎたい」)、それを次セッションの焦点として文書を調整する。

### 2. 引き継ぎ文書を書く

下の「文書テンプレート」に沿って書く。原則:

- **続きに必要な情報だけ**書く。会話の全文を写経しない。
- **すでに別ファイルにある内容は重複させない**。計画・メモ・既存のドキュメントなどは、本文に貼らずパスやファイル名で参照する。
- **機微情報は伏せる**。パスワード、APIキー、口座番号などは書かない。具体的な金額や個人情報は、引き継ぎに必要な範囲にとどめる。
- 引き継ぎ先が**最初に読むべきファイル**があれば明示する(関連するメモやドキュメントなど)。

### 3. 保存する

- 保存先: 引き継ぎ文書をまとめて置くフォルダ(例: 作業フォルダ内の `handoffs/` など。無ければ作る)。
- ファイル名: `YYYY-MM-DD_引き継ぎ元→引き継ぎ先_要件.md`(例: `2026-06-03_健康→家計_収入変動後の試算.md`)
- 日付は bash の `date +%F` で確定させる。

### 4. 報告する

保存したファイルを提示し、引き継ぎ先での始め方を一言添える(例: 「新しいチャットで『この引き継ぎ文書を読んで続きをやって』と言えばOK」)。

## 引き継ぎ後の運用ルール(重要)

**引き継ぎ文書は使い捨て。** 受け取った側が内容を読み込んで作業を引き継いだら、その引き継ぎ文書は削除する。これにより引き継ぎフォルダが古い文書で荒れない。

- 引き継ぎ先のセッションで「この引き継ぎ文書を読んで」と言われたら、内容を反映したあと、残すべき恒久的な事実は自分のメモやノートに書き、引き継ぎ文書本体は削除する。
- 削除前に「引き継ぎ完了。この引き継ぎ文書は削除していい?」と一言確認してもよい。

## 文書テンプレート

```markdown
# 引き継ぎ: [引き継ぎ元の話題] → [引き継ぎ先の話題]

- 作成日: YYYY-MM-DD
- 次のセッションの目的: [何にフォーカスするか。1〜2文]

## 引き継ぎ先がまず読むべきファイル
- [パス/ファイル名] — [なぜ読むか]

## これまでの経緯(要点)
[なぜこの話になったか。続きに必要な背景だけを数行で]

## いま分かっていること / 決まったこと
- [事実・前提・数値など。続きの判断に効くものだけ]

## 引き継ぎ先にやってほしいこと
- [具体的なタスク。曖昧さを残さない]

## 未確定・要確認
- [確認すべき点、まだ決まっていない論点]
```
