# farman.jp お知らせ一括登録

farman.jp のWordPress「お知らせ」に、複数の記事をまとめて登録するための手順。
PR TIMESのプレスリリース、テレビ放映、メディア掲載、講演・登壇、イベント参加などを、
既存記事のトーン・書式に合わせて投稿する。

## 前提の確認

- 操作は **Claude-in-Chrome MCP**（`mcp__claude-in-chrome__*`）で行う。ツールが未ロードなら ToolSearch でまとめて読み込む。
- ユーザーの Chrome で **farman.jp のWordPress管理画面にログイン済み**であること（`admin-Farman`）。
  未ログインなら、ユーザーにログインを依頼する（パスワード入力は代行しない）。
- 「お知らせ」は**固定ページではなく投稿(posts)**。一覧は `https://farman.jp/wp/wp-admin/edit.php`。
- サイト構造・カテゴリID・REST APIの詳細・投稿用JSテンプレートは
  **`references/site-details.md` を必ず読む**こと。ここが作業の核心。

## 全体の流れ

1. **既存記事を1〜2件確認**して現在の書式を把握する（初回や不明時）。
   REST で `GET /wp-json/wp/v2/posts/{id}?context=edit&_fields=content` すれば本文HTMLが取れる。
2. **入力を分類する**。各項目を次のどれかに振り分ける（→カテゴリと書式が決まる）:
   - TV放映 → メディア。有吉ゼミ系は既存テンプレ（画像＋本文）を流用。
   - プレスリリース/報道/掲載/寄稿/動画公開 → メディア。
   - 講演・登壇・イベント協力・審議会委員 → 活動報告。
   - 一般的なお知らせ → お知らせ。
3. **各URLから情報を取得**する（タイトル・日付・要約・ファーマンとの関係）。
   取得方法と、取得できないサイトの回避策は `references/fetching.md` を参照。
4. **タイトルと本文を組み立てる**。命名規則・本文フォーマットは `references/site-details.md`。
   タイトルは媒体/主催者を【】で囲み、弊社視点（「弊社代表井上が登壇しました」等）で書く。
   本文は〈短い説明文＋出典リンク〉が基本。
   **画像（アイキャッチ＋本文画像）を入れる場合は `references/images.md` を読む**。
   各記事の元ページから素材を拾い、REST の media にアップロードして featured_media と
   本文先頭の `<img>`(幅400pxで統一)に設定する。CORSで直接取得できないホストは
   画像プロキシ wsrv.nl を噛ませる、素材が無い記事は既存の井上写真(ID169)をフォールバック。
5. **作成前に方針をすり合わせる**。一覧表（#/日付/カテゴリ/タイトル案）を提示し:
   - 投稿日を元記事の日付に合わせるか、本日にするか
   - 既存記事との重複がないか（同じイベントの二重掲載など）
   - 日付が取れなかった項目の扱い
   ※**公開状態は毎回聞かなくてよい**。このスキルは常に「まず下書きで作成 → 完成後にチェックを投げる」
   フロー（下記6〜8）で進めるのが既定。
6. **必ず下書き(status:'draft')で作成し、画像まで仕上げる**。`references/site-details.md` の
   JSテンプレートを wp-admin タブの `javascript_tool` で実行。まず1件テスト → 残りを一括、が安全。
   続けて `references/images.md` に従い、各記事にアイキャッチ＋本文画像(幅400px)を設定する。
   `GET /wp-json/wp/v2/posts?status=draft&_fields=id,featured_media,content` で
   全件に featured_media と本文 `<img` が入っているか検証する。
7. **【必須チェックポイント】下書き完成段階で、いったんユーザーにチェックを投げる。**
   ここで作業を止め、勝手に公開しない。ユーザーに次を渡してレビューを依頼する:
   - 下書き一覧URL `https://farman.jp/wp/wp-admin/edit.php?post_status=draft&post_type=post`
   - 作成した各記事の一覧（日付/カテゴリ/タイトル）と、画像の入手元・要確認点
     （報道機関の写真を使った項目、フォールバック画像を使った項目、日付不明の項目など）
   そのうえで「内容を確認のうえ、公開してよいかお知らせください」と明示的に承認を求める。
8. **ユーザーの公開OKが出てから公開する**（`references/site-details.md` の「まとめて公開」JS）。
   修正指示があれば下書きを直し、再度チェックを投げる。承認前に publish しないこと。

## 大事な注意（ハマりどころ）

- **REST APIのルートは `https://farman.jp/wp-json/`**。サイトは `/wp/` サブディレクトリだが
  APIは `/wp-json/` 直下。`/wp/wp-json/` は404になる。
- **`javascript_tool` の戻り値に `?...` を含むURLを入れない**。ブラウザ拡張のフィルタが
  `[BLOCKED: Cookie/query string data]` で表示を潰す。戻り値は id / title / date / status など
  クエリ文字列を含まない値だけにする（POST本文にURLを入れるのは問題ない）。POST後は
  下書き一覧を別途 GET して結果を確認する。
- **公開は必ずユーザーのチェック・承認後**。作成は必ず draft で行い、下書き完成段階で
  一度ユーザーにレビューを投げる（ステップ7）。承認前に status:'publish' にしない。
- **投稿日**は `date:'YYYY-MM-DDT10:00:00'` の形式で指定。元記事の日付に合わせると
  お知らせ一覧が時系列で正しく並ぶ。下書きに過去日を設定 → 公開時もその日付になる。

## 出力（ユーザーへの報告）

- **下書き完成時（ステップ7・チェックポイント）**: 作成件数、カテゴリ内訳、下書き一覧URL
  （`https://farman.jp/wp/wp-admin/edit.php?post_status=draft&post_type=post`）、
  各記事の日付/カテゴリ/タイトル、画像の入手元と要確認点（報道写真・フォールバック・日付不明など）を
  提示し、**「確認して公開OKか教えてください」と承認を求めて一旦停止**する。
- **公開後**: 公開件数、公開ページURL（`https://farman.jp/news/`）、残課題を報告する。
