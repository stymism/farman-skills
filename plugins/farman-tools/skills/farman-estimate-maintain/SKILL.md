# FARMAN見積システム 改修・保守ガイド

有機野菜の見積もりWebアプリ「FARMAN_Estimate system」を安全に改修するための手順とハマりどころ。
サーバー（Google Apps Script）とクライアント（1枚のHTML）の2ファイル構成で、自社HPに **iframe埋め込み** して公開する。

まず「これから何を変えるのか」を1文で確定してから着手すること。多くの改修は
「マスタ（スプレッドシート）を変える」だけで済み、コードを触る必要がないことも多い（下記を参照）。

## システムの全体像

- **正体**: GAS Web App（匿名公開・`executeAs: USER_DEPLOYING` / `access: ANYONE_ANONYMOUS`）。自社HPに `<iframe>` で載せる
- **3つのフロー**: 通常配送（toC・小口宅配）/ 大口注文（bulk・トラック単位）/ 体験メニュー（experience・農業体験）
- **データの持ち方**: 品目・送料・エリア・体験メニューは**すべてスプレッドシートの各マスタシート**が正。コードにハードコードしない（フォールバック用のデフォルトのみコード内にある）
- **問い合わせの記録**: 送信されるとフロー別のリードシートに1行追記＋顧客宛/スタッフ宛メールを送信
- スプレッドシートID・宛先などの定数は `コード.js` 冒頭にまとまっている

詳細なデータ構造（各マスタの列定義・計算ロジック・受付番号／スパム対策の内部・キャッシュ）は
**`references/architecture.md` を読む**こと。マスタの列を足す・計算式を変える等はここが核心。

## ファイルの場所と「正本」ルール

- ローカルの作業コピー（現状 `C:\Users\farma\Downloads\` にある）:
  - `コード.js` — サーバー側の全ロジック（GASの `Code.gs` に対応）
  - `index.html` — クライアント全体（CSS/HTML/JSが1ファイル。GASの `index.html` に対応）
  - `appsscript.json` — マニフェスト（スコープ3つ: spreadsheets / script.scriptapp / script.send_mail）
- **ローカルファイルが正**。ユーザーが手動でGASエディタにコピペ→再デプロイして反映する運用。
  過去に「編集したのに反映していなかった」事故があるので、**変更後は必ず「GASへ貼り直して再デプロイしてください」と明示的に伝える**こと。

## 改修前に必ず思い出す4つのハマりどころ

過去に実際にハマった／壊れた箇所。改修に入る前にこの4点を頭に入れておくと事故が防げる。

1. **`index.html` の `initMasterData()` はスクリプト最末尾から動かさない。**
   このIIFEは埋め込みマスタJSONを即時反映するが、`buildTocTable()` が参照する
   `let tocGrouped = {}` などの宣言より**前**に置くと、`let` の一時的死角（TDZ）で
   `ReferenceError: Cannot access 'tocGrouped' before initialization` になり、
   **初回ロード画面（スピナー）から一切進まなくなる**。呼び出しは必ず全ての `let`／関数定義の後（最末尾）に置く。

2. **リードシートのヘッダー（列）を変えたら `migrateLeadsSheet()` を1回実行してもらう。**
   `コード.js` の `LEAD_SHEET_INFO` のヘッダー配列を編集したら、GASエディタで
   `migrateLeadsSheet()` を実行するようユーザーに依頼する。列名でデータを引き当て直すので
   **既存データは安全・何度実行してもOK**（旧ヘッダーに無い新列は空欄で埋まる）。これを忘れると
   新しい列と既存データがズレる。

3. **デプロイのたびに `APP_VERSION` を上げる（現在の値はコードで確認）。**
   `コード.js` 冒頭の `APP_VERSION` を上げてから再デプロイすると、フッターに `v1.x.x` が出る。
   これは**反映できたかを一目で確認する仕組み**。数字が変わっていなければ反映漏れ＝古いまま動いている。

4. **`index.html` はGASテンプレートとして評価される前提。ローカル直開きでは動かない（仕様）。**
   `doGet` が `HtmlService.createTemplateFromFile('index')` で読み込み、
   `<?!= masterDataJson ?>`（マスタJSON埋め込み）と `<?!= appVersion ?>`（版数）を差し込む。
   このスクリプトレットがあるため、ファイルをブラウザで直接開いても動かず、ガイドメッセージが出る。
   **GASエディタ上のHTMLファイル名は必ず `index`**（`createTemplateFromFile('index')` が名前で参照している）。

## 「コードを触らずマスタだけで済む」改修

以下はスプレッドシートのマスタシートを編集するだけで反映される（コード変更・再デプロイ不要。
反映は最大6分のキャッシュ後、または `?cache=clear` を付けてアクセス＋スーパーリロードで即時）:

- 品目の追加・単価変更・販売時期の変更・有効/無効の切替（品目マスタ_通常配送 / 品目マスタ_大口注文）
- 送料の変更（送料マスタ_通常配送）、エリアの増減（エリアマスタ_通常配送）
- 体験メニューの増減（体験メニューマスタ）

各マスタの列の意味は `references/architecture.md` を参照。**マスタを足すときは有効フラグ列を忘れない**。

## 画面遷移（ナビゲーション）の設計

3フロー（toc 4ステップ / bulk 6ステップ / exp 2ステップ）で**挙動を必ず揃える**こと。
ステップを増減する改修をしたときは、下記が3フローとも成立しているかを確認する。

- **ステップバーは完了済みだけクリックで戻れる**。各ステップ画面のインジケーターHTMLが
  `.done`（完了）/ `.active`（現在地）/ クラス無し（未到達）を静的に持っており、
  `initStepNavigation()` が `.done` にだけ `role=button` とクリック／Enterキーのハンドラを付ける。
  **未到達の段階を押せるようにしてはいけない**（データ未入力のままサーバー計算に飛んでエラーになる）。
  ステップを追加したら、各画面のインジケーターHTMLの `done`/`active` の付き方が正しいか要確認。
- **どこからでもトップへ戻れる**。ヘッダーは `position:sticky` なので常に見えている。
  ヘッダーのロゴ（`.hd-brand`）と、フロー中だけ表示される `#hd-home-btn`（「← 最初に戻る」）の
  両方が `showScreen('top')` を呼ぶ。`showScreen()` がこのボタンの表示/非表示を切り替える。
- **フローに入り直したら必ずステップ1から**。`showScreen()` は `name !== 'top'` のとき
  `showStep(name, 1)` を呼ぶ。ここを特定フローだけ除外すると挙動がズレる
  （実際に exp だけ除外されていてステップ2に留まる不具合があった）。
  入力値は `state` とDOMに残るため、ステップ1に戻しても再入力にはならない。
- 狭い画面ではヘッダーのタグライン `.hd-jas` を `display:none` にして「最初に戻る」の場所を確保している。
- **画面遷移の関数を新設したら、末尾に `if (window.__reportHeight) window.__reportHeight();` を足す。**
  iframe埋め込み時に高さを追従させるため（詳細は `references/iframe-embed.md`）。忘れると
  その画面だけ余白やiframe内スクロールが出る。
- **画面内スクロールは `window.scrollTo` ではなく `scrollAppTo()` を使う。**
  高さ自動調整でiframe内にスクロール余地が無くなり、埋め込み時は `window.scrollTo` が
  効かない（品目選択後に小計まで移動しない等）。`scrollAppTo()` が親ページへ委譲する。
- **行き止まりを作らない。「進めない」状態には必ず「戻って直す」導線を添える。**
  toC見積画面（step3）で最低金額未満のとき、押せない「問い合わせフォームへ進む」を
  `hidden` にし、代わりに `#toc-add-items`（「＋ 品目を追加する」→ `showStep('toc',1)`）を
  出している。`renderTocEstimate` のハンドラが `est.canSubmit` で両ボタンを出し分ける
  （`classList.toggle('hidden', …)`）。押せないボタンだけ残すと手詰まりに見えるため。
  選択済みの品目は `state.toc.qtys` に残るのでステップ1へ戻っても再入力にならない。

## UXの実装済みパターン（v1.5.0・崩さない）

- **判定に使う数字は入力中の画面で見せる。** 最低注文金額の残額は品目選択画面の
  `#toc-min-progress` に常時表示（ゼロ=非表示/未満=warn「あと¥○○」/以上=ok）。
  金額はハードコードせず `masterData.minOrderAmount`（サーバー定数）から表示。
  後の画面で「足りません」と差し戻す作りにすると往復が発生する。
- **選択肢が十数個を超えるものをプルダウンにしない。** 大口の数量（0.1t刻み・最大60通り）は
  `#bulk-qty-input` のステッパー（−/＋＋直接入力）。範囲・刻みは input の `dataset` に持たせ、
  `bulkQtyChange`（クランプ）と `bulkGoStep4`（丸め・範囲検証）が参照する。
- **入力エラーはボタンのそばに出す。** 各フォームにインラインのエラー要素
  （`#toc-step2-error` / `#toc-submit-error` / `#bulk-submit-error` / `#exp-submit-error` /
  `#bulk-qty-error`）。`showInlineError`（上部バナー＋先頭スクロール）は
  「前の画面に戻す計算エラー」専用。入力位置から引き剥がさない。
- **送信完了は完了画面で示す。** `showSubmitComplete(flow, receiptNo)` がフォーム本体
  （`#○○-form-body`）を畳み、タイトルを差し替え、受付番号＋「最初の画面に戻る」だけにする。
  **対になる `resetSubmitComplete(flow)` を `showScreen` が再入場時に必ず呼ぶ**
  （呼ばないと2件目が送れない不具合になる。実際に起きた）。
- **一覧カードには判断材料を載せる。** 品目カードは販売中規格の最低税込価格
  「2種・¥324〜」を表示（単一規格なら「〜」なし）。開かないと値段が分からない状態にしない。

## コードを変更するときの手順

1. **変更対象を特定**する。サーバー計算・メール・リード保存・採番・スパム対策なら `コード.js`、
   画面・入力フォーム・見た目・バリデーション表示なら `index.html`。両方に関わる変更（新しい項目を
   フォームに足してリードシートにも保存する等）は、送信データの受け渡し（クライアントの `data` オブジェクト →
   `submitLead` が読むフィールド名）の整合を必ず合わせる。
2. **編集後、構文チェック**する（GASに貼る前にローカルで壊れを検知できる）:
   ```bash
   node --check コード.js
   ```
   `index.html` は `<script>...</script>` の中身を抜き出し、テンプレート変数を仮値に置換してから
   `node --check` する（`<?!= masterDataJson ?>` → `null`、`<?!= appVersion ?>` → `""`）。
3. **ローカルで実際に動かして確認する**（UI・遷移まわりの改修では特に有効）。
   `index.html` はそのままでは動かないが（テンプレート構文＋`google.script.run` が無い）、
   **検証用コピーを作れば実挙動をブラウザで確認できる**:
   - `<?!= masterDataJson ?>` をダミーのマスタJSONに、`<?!= appVersion ?>` を仮の版数に置換
   - `google.script.run` のモックを `</body>` 直前に差し込む
     （`withSuccessHandler`/`withFailureHandler` が自身を返し、各メソッドが `setTimeout` で
     代表的なレスポンスを返すだけの簡易実装で足りる）
   - `_uitest.html` 等の名前で Downloads に置き、launch.json の `farman-estimate`
     （ポート8931・Downloadsを配信）で開く。**確認後は必ず削除する**
   遷移の検証は画面を目視するより、`showScreen()`/`showStep()` を呼んでから
   「どのステップが `hidden` でないか」「押せる段階はどれか（`role=button` の有無）」を
   まとめて出力させたほうが速く、3フローの取りこぼしにも気づける。
4. **ユーザーに反映を依頼**する。デプロイ手順は下記チェックリスト。

## デプロイ・反映チェックリスト（ユーザーに渡す手順）

1. `コード.js` の内容をGASエディタの `Code.gs` に貼り直す
2. `index.html` の内容をGASエディタの `index`（HTMLファイル）に貼り直す
3. （コード側を変えたなら）`APP_VERSION` を1つ上げたことを確認
4. 「デプロイ」→「デプロイを管理」→ 既存のデプロイを編集 → **バージョンを「新バージョン」にして**デプロイ
   （※「新バージョン」にしないとURLは同じでも中身が古いまま。ここが最頻の反映漏れ原因）
5. リードシートのヘッダーを変えたなら `migrateLeadsSheet()` を実行
6. iframe（自社HP）を再読み込みし、フッターの `v` が上げた番号になっているか確認

## iframe埋め込みの注意点

自社HPへの埋め込みで実務上効いてくる点。詳細と推奨タグは **`references/iframe-embed.md`** を参照。要点だけ:

- `コード.js` の `doGet` に `setXFrameOptionsMode(HtmlService.XFrameOptionsMode.ALLOWALL)` が必要（埋め込み許可）。既に入っている。消さない
- GASは中身を自前のiframeで包むため、自社HPのiframeと合わせて**二重iframe**になる。高さが自動フィットしないので
  **iframe側に十分な高さ（目安1000〜1200px）を固定指定**する。ステップ遷移時の `window.scrollTo` は
  内側フレーム内で完結し親ページはスクロールしない＝高さを取っておけば違和感が出ない
- コンソールの `allow-scripts and allow-same-origin...` 警告はGoogle側由来で無害

## 変更してはいけない／特に慎重に扱うもの

- `appsscript.json` のスコープを減らさない（メール送信・トリガー・シート操作に必要）
- リード保存前のサーバー側再検証（`submitLead` 内の必須チェック・メール形式・最低金額・レート制限）は
  スパム／不正送信の防波堤。安易に外さない
- 1回きり実行の関数（`createExpMenuMasterSheet` 等・実行済み）を再実行しても基本安全な設計だが、
  意図せず実行しないよう、改修対象と混同しないこと
