## 1. 主要定数（コード.js 冒頭）

| 定数 | 意味 |
|---|---|
| `SPREADSHEET_ID` | マスタ・リードを保持するスプレッドシート |
| `CONTACT_EMAILS` | スタッフ宛通知メールの宛先（配列） |
| `MIN_ORDER_AMOUNT` = 15000 | toC最低注文金額（品目+送料、税込）。これ未満は送信不可 |
| `TAX_RATE` = 0.08 | 青果の軽減税率8%固定 |
| `TAX_RATE_SHIP` = 0.10 | 送料の標準税率10%（**大口の税抜送料にのみ適用**。toCの送料マスタは税込なので対象外） |
| `BOX_SIZE_MAX` = 10000 | サイズFIX 1万ごとに1箱（〜10000で1箱、10001〜で2箱…） |
| `BOX_MAX` = 13 | 最大箱数（サイズFIX合算13万まで）。超過は大口へ誘導 |
| `COOL_FEE_PER_BOX` = 715 | クール便の箱単価加算（税込）。クライアントはこの値を`coolFeePerBox`で受け取り動的表示 |
| `CACHE_KEY_V2` / `CACHE_SEC`=360 | サーバー側マスタキャッシュのキーと秒数（6分） |
| `APP_VERSION` | 反映確認用の版数。**デプロイのたびに手動で上げる** |
| `DAILY_SUBMIT_LIMIT` = 200 | 1日あたりのリード受付上限（メール日次上限の枯渇防止） |
| `RECEIPT_PREFIX` | `{ toC:'T', bulk:'B', experience:'E' }` 受付番号のフロー別接頭辞 |
| `LEAD_STATUS_OPTIONS` | 対応状況プルダウンの候補（種別ごと）。変更後 `migrateLeadsSheet()` で反映 |
| `MAIL_SIGNATURE` | 顧客宛メール末尾の署名（配列）。行を足せば電話・住所等を追加できる |

## 2. スプレッドシートのマスタ列定義

いずれも**1行目がヘッダー、有効フラグが0の行は除外**される。列は位置（A,B,C…）で読むので**列の挿入順に注意**。

**品目マスタ_通常配送**（`_readItemsToC`）
`A=ID / B=カテゴリ / C=品目名 / D=規格・単位 / E=単価(税抜) / F=サイズ基準値 / G=サイズ係数 / H=クール便推奨 / I=有効フラグ / J=販売開始(MM/DD) / K=販売終了(MM/DD)`
- サイズFIX = F×G（コード側で算出。シートには持たない）
- 販売時期: MM/DD形式・年なし。開始>終了は年跨ぎ（例 08/01〜03/31）。未設定は通年。期間外の品目は選択不可表示＋サーバー側でも除外
- カテゴリはクライアントのタブ（葉物類/根菜類/果菜類）と一致させる

**品目マスタ_大口注文**（`_readItemsBulk`）
`A=品目名 / B=単価(円/t・税抜) / C=最小ロット(t) / D=刻み(t) / E=中型上限t / F=大型上限t / G=関東中型送料 / H=関東大型送料 / I=関西中型送料 / J=関西大型送料 / K=有効フラグ`
- 送料も税抜。税込換算は `calcEstimateBulk` で行う
- 中型=最小ロット〜中型上限、大型=（中型上限+刻み）〜大型上限。数量帯が重複しないよう自動算出

**送料マスタ_通常配送**（`_readFareToC`）: `A=エリア名 / B=箱あたり送料(税込)`
**エリアマスタ_通常配送**（`_readAreas`）: `A=エリアID / B=エリア名 / C=表示名 / D=有効フラグ`（表示名がプルダウンに出る）
**体験メニューマスタ**（`_readExpMenus`）: `A=ID / B=メニュー名 / C=有効フラグ`（「その他」はマスタに置かずクライアント固定で末尾に出る）

いずれも読めない場合は `_defaultXxx()` のフォールバックが返る。フォールバックは非常用で、正はシート。

## 3. リードシート（保存先）とヘッダー

`LEAD_SHEET_INFO` にフロー別のシート名とヘッダー配列が定義されている。**ここを編集したら必ず `migrateLeadsSheet()`**。
現行ヘッダー（先頭が受付番号／末尾から2列目が対応状況「未対応」自動セット／末尾が社内メモ空欄）:

- `リード_通常配送`: 受付番号/日時/氏名/会社名・店舗名/メール/電話/配送先住所/備考/エリア/クール便/箱数/品目小計/送料/合計/品目明細/対応状況/社内メモ
- `リード_大口注文`: 受付番号/日時/氏名/会社名・店舗名/メール/電話/配送先住所/備考/品目/トラック/数量t/地域/品代/送料/合計/対応状況/社内メモ
- `リード_体験メニュー`: 受付番号/日時/氏名/会社名・店舗名/メール/電話/備考/体験メニュー/人数/希望日程/場所/お弁当・飲み物/移動手段/対応状況/社内メモ

- **「備考」は顧客入力、「社内メモ」はスタッフ用の自由記入欄**。両者は役割が違うので別名。
  **ヘッダー名を重複させてはいけない**（`_syncLeadSheet` が列名をキーに引き当てるため、
  同名列があるとデータがズレる）。列を足すときは既存ヘッダーと重複しない名前にする。
- **対応状況はプルダウン**（データ検証）。候補は `LEAD_STATUS_OPTIONS`（種別ごと）:
  toC/bulk=未対応/対応中/受注確定/発送済/完了、experience=未対応/対応中/受注確定/完了。
  `_statusRule(type)` が規則を作り、`_applyStatusValidation`（既存行一括）と
  `_applyStatusValidationToRow`（新規1行）で貼る。`_syncLeadSheet` はヘッダー一致時でも
  毎回プルダウンを貼り直す（候補を変えて `migrateLeadsSheet()` すれば反映される）。
  `allowInvalid=true`（想定外値を弾かない・矢印は出す）。
- `_s()` で数式インジェクション対策（先頭が `= + - @ |` の値はクォートで無害化）してからセルに入れている。
  新しく自由入力値を列に足すときも `_s()` を通すこと。

## 4. データの流れ（初回ロード高速化の仕組み）

以前はページ表示後に `google.script.run.getMasterData()` を呼んでいたため初回10〜15秒かかっていた。
現在は **`doGet` がマスタJSONをHTMLに直接埋め込む**方式:

1. `doGet` → `createTemplateFromFile('index')` → `tpl.masterDataJson = JSON.stringify(getMasterData()).replace(/</g,'\\u003c')`
   （`<` をエスケープするのは、品目名等に `</script>` 相当が入ってもタグを閉じさせない安全対策。
   スクリプトレットは `<?!= ?>`＝**非エスケープ出力**を使う。JSONをHTMLエンティティ化させないため）
2. `index.html` 冒頭 `const EMBEDDED_MASTER = <?!= masterDataJson ?>;`
3. スクリプト最末尾の `initMasterData()` が `EMBEDDED_MASTER` を `applyMasterData()` に渡して即描画
4. localStorageキャッシュは廃止（埋め込みデータが常に最新のため不要）

**注意**: `initMasterData()` を宣言より前に置くとTDZエラーで全停止（SKILL.mdのハマりどころ1）。

## 5. 見積計算ロジック

**toC（`calcEstimateToC`）**: 選択品目の税込単価×数量を合計→サイズFIX合計を1万で割って切り上げが箱数（上限13）→
`送料 = (エリア箱単価 + クール便715×該当) × 箱数` → 合計が `MIN_ORDER_AMOUNT` 以上で送信可。
エリア名がマスタに無ければ `AREA_NOT_FOUND`（古いキャッシュ対策）。販売時期外の品目は除外して `offSeason` に入れて返す。

**大口（`calcEstimateBulk`）**: `品代税抜 = 単価×t` を8%税込に、送料は税抜マスタ値を10%税込に換算。
関東/関西以外、または送料未設定は `fareNote` で「別途」。最小ロット未満は `MIN_LOT_NOT_MET`。

クライアントにも同じ箱数・最小ロットのロジックがある（`updateTocSummary` / `bulkMinLotForTruck`）。
**計算式を変えるときはサーバーとクライアントの両方を合わせる**こと（サーバーが正・最終値）。

## 6. 受付番号の採番（`_generateReceiptNo`）

`接頭辞-yyyyMMdd-連番3桁`（例 `T-20260721-001`）。`LockService` で排他制御し、日次連番を
`PropertiesService`（ScriptProperties）の `receipt_seq_<接頭辞>_<日付>` に保持。接頭辞は `RECEIPT_PREFIX`。

## 7. スパム・ボット対策

- **honeypot**: フォームに不可視の `website` 入力（`toc-hp`/`bulk-hp`/`exp-hp`）。ボットが埋めると
  `submitLead` は保存・送信せず**偽の成功**を返す（気づかせない）
- **レート制限**（`_checkRateLimit`）: 同一メールを60秒に1回まで（CacheService）
- **日次上限**（`_checkDailyQuota`）: 1日 `DAILY_SUBMIT_LIMIT` 件まで（ScriptProperties）
- **サーバー再検証**: 必須項目・メール形式（`EMAIL_RE`）・toC最低金額・住所必須をサーバーでも再チェック

## 8. キャッシュ

`getMasterData()` は `CacheService`（スクリプトキャッシュ・6分）を使う。マスタ変更の即時確認は
Web App URLに `?cache=clear` を付けてアクセス→ブラウザをスーパーリロード。`refreshCache()` は毎朝4時に
`setupTrigger()` が仕込んだトリガーでも走る。

## 9. メール送信（`_sendLeadMails`）

顧客宛（受付番号＋見積内容）とスタッフ宛（`CONTACT_EMAILS`・連絡先＋見積明細）の2通。
見積明細は `_estimateDetailLines()` が顧客/スタッフ共通で生成。件名にも受付番号が入る。
スタッフ宛は `replyTo` にお客様アドレスを入れているのでそのまま返信できる。
顧客宛は見積の下に**「担当者より改めてご連絡いたします」の結び＋署名**が付く。署名は
`MAIL_SIGNATURE`（配列）で管理。**電話・住所を入れる場合はこの配列に行を足す**
（勝手に作らず本人に確認した実在の連絡先を入れること）。大口のみ「50%前払い」の注記が残る。

## 10. 1回きり実行／トリガー系の関数（GASエディタから手動実行）

| 関数 | 用途 | 再実行 |
|---|---|---|
| `migrateLeadsSheet()` | リードシートを種別別に用意＋ヘッダー同期＋対応状況プルダウン設定 | 何度でも安全 |
| `createExpMenuMasterSheet()` | 体験メニューマスタを新規作成 | 既存なら何もしない |
| `setupTrigger()` | 毎朝4時の `refreshCache` トリガー設定 | 重複作成しない設計 |
| `refreshCache()` | キャッシュ破棄→再構築 | いつでも安全 |
